# 城市三维可视化与分析平台实现说明

本文说明该项目的整体实现方式，重点解释后端架构、前端架构，以及前端如何与后端交互。

## 1. 项目整体结构

项目是一个前后端分离的三维 WebGIS 应用：

- 前端：`Vue 3 + Vite + CesiumJS + Axios`
- 后端：`FastAPI + Pydantic + NumPy/SciPy + Shapely + NetworkX`
- 数据格式：前后端主要通过 `GeoJSON` 交换空间数据
- 开发期通信：前端请求 `/api/...`，由 Vite 代理转发到 `http://localhost:8000`

核心目录如下：

```text
urban-3d-gis/
├── frontend/                 # Vue + Cesium 前端
│   ├── src/
│   │   ├── api/              # Axios 后端接口封装
│   │   ├── cesium/           # Cesium 场景、图层渲染、三维分析
│   │   ├── components/       # 控制面板、结果面板、AI 助手
│   │   ├── App.vue           # 前端主流程入口
│   │   └── main.js
│   └── vite.config.js        # Vite 插件与代理配置
├── backend/
│   ├── app/
│   │   ├── main.py           # FastAPI 应用入口
│   │   ├── config.py         # 研究区与运行配置
│   │   ├── data_loader.py    # GeoJSON 数据加载与样例数据回退
│   │   ├── routers/api.py    # REST API 路由
│   │   └── engine/           # 空间分析引擎
│   └── requirements.txt
└── data/                     # 真实抓取数据或样例数据来源
```

## 2. 后端实现

后端入口是 `backend/app/main.py`。它创建 FastAPI 应用，启用 CORS，并挂载 `backend/app/routers/api.py` 中定义的 API 路由。

```python
app = FastAPI(...)
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)
app.include_router(router)
```

后端的职责可以分为三层：

1. API 层：接收 HTTP 请求、校验参数、返回 JSON/GeoJSON。
2. 数据层：加载 POI、道路、避难场所、水系等 GeoJSON 数据。
3. 分析层：执行价值评估、路径规划、热点分析、洪水模拟、AI 调度等算法。

### 2.1 API 路由层

所有业务接口定义在 `backend/app/routers/api.py`，统一使用 `/api` 前缀：

```python
router = APIRouter(prefix="/api")
```

接口按功能分组：

| 功能 | 接口 | 方法 | 说明 |
|---|---|---|---|
| 研究区元数据 | `/api/cities` | GET | 返回城市配置、默认城市、POI 中文名 |
| POI 数据 | `/api/data/pois` | GET | 返回当前城市 POI |
| 道路数据 | `/api/data/roads` | GET | 返回当前城市路网 |
| 避难场所 | `/api/data/shelters` | GET | 返回避难场所点 |
| 地段价值评估 | `/api/analysis/value` | POST | 返回价值网格 GeoJSON |
| 选址分析 | `/api/analysis/site` | POST | 返回高分地块 GeoJSON |
| 路径规划 | `/api/route` | POST | 返回路线 LineString |
| 撤离路径 | `/api/evacuate` | POST | 返回撤离路线与避难场所 |
| 热点分析 | `/api/stats/hotspot` | POST | 返回带热点分类的网格 |
| 服务区/等时圈 | `/api/service-area` | POST | 返回等时圈 Polygon |
| 洪水模拟 | `/api/flood` | POST | 返回淹没面和危险区 |
| 洪水动画 | `/api/flood/animation` | POST | 返回多帧淹没结果 |
| AI 助手 | `/api/ai` | POST | 自然语言转分析调用 |
| 重置 AI 会话 | `/api/ai/reset` | POST | 清空某个会话上下文 |

请求体使用 Pydantic 模型定义，例如路径规划：

```python
class RouteReq(BaseModel):
    city: str = DEFAULT_CITY
    start: list[float]
    end: list[float]
    optimize: str = "time"
    mode: str = "drive"
    vias: list[list[float]] | None = None
    hazard: dict | None = None
```

这样后端可以自动校验请求字段，并在 `/docs` 中生成接口文档。

### 2.2 配置层

`backend/app/config.py` 保存研究区元数据：

- `center`：Cesium 相机飞行中心
- `bbox`：分析网格和样例数据生成范围
- `camera_height`：默认相机高度
- `utm_epsg`：用于距离计算的投影坐标系

示例：

```python
"hangzhou_core": {
    "name": "杭州·主城区",
    "center": [120.16, 30.27],
    "bbox": [120.00, 30.13, 120.42, 30.42],
    "camera_height": 16000,
    "utm_epsg": 32650,
}
```

DeepSeek 配置也在这里读取：

```python
deepseek_api_key: str = ""
deepseek_base_url: str = "https://api.deepseek.com"
deepseek_model: str = "deepseek-chat"
```

如果 `.env` 没有配置 `DEEPSEEK_API_KEY`，AI 助手会自动使用本地 mock 规则引擎。

### 2.3 数据加载层

`backend/app/data_loader.py` 负责加载空间数据。它优先读取 `data/<city>/` 下的真实 GeoJSON：

- `pois.geojson`
- `roads.geojson`
- `shelters.geojson`
- `water.geojson`
- `dem.npy`

如果文件不存在，或者文件存在但要素为空，则回退到 `sample_data.generate_all(city)` 生成样例数据。这样项目即使没有真实 OSM 数据，也能离线启动和演示。

数据加载函数使用 `@lru_cache` 做内存缓存，避免每次请求重复读盘：

```python
@lru_cache(maxsize=8)
def load_pois(city: str) -> dict[str, Any]:
    return _load_or_generate(city, "pois.geojson", "pois")
```

### 2.4 空间分析引擎

后端算法集中在 `backend/app/engine/`。

#### 地段价值评估：`value.py`

核心思想是“缓冲区 + 距离衰减 + 加权叠加”：

1. 在研究区 bbox 内生成规则网格。
2. 对每个网格中心，计算到景点、商业区、学校、医院、公交、主干道等 POI 的最近距离。
3. 使用指数衰减函数将距离转换为邻近度得分：

```text
s = exp(-d / decay)
```

4. 按权重加权求和，再归一化到 0 到 100。
5. 返回 `FeatureCollection`，每个网格是一个 Polygon，`properties.score` 是综合分数。

选址分析复用价值评估结果，只保留 `score >= min_score` 的网格，并可按 `top_k` 截断。

#### 路径规划：`routing.py`

路径规划把道路 GeoJSON 转成 NetworkX 图：

- 节点：道路折点，经纬度取 6 位小数后作为节点 key。
- 边：相邻折点之间的道路段。
- 边属性：长度 `length`、行驶时间 `time`、速度 `speed`。

查询流程：

1. 将用户点击的起点、终点吸附到最近路网节点。
2. 根据 `optimize` 选择权重：
   - `time`：最短时间
   - `length`：最短距离
3. 使用 `networkx.shortest_path` 求路径。
4. 将节点路径转换为 GeoJSON `LineString` 返回。

撤离路径会遍历避难场所，选择从起点可达且耗时最短的避难点。如果传入洪水危险区 `hazard`，后端会先删除与危险多边形相交的道路边，再重新计算路径。

#### 热点分析：`stats.py`

热点分析先调用价值评估得到网格，再基于指定字段计算空间统计。默认字段是综合分数 `score`，也可以切换到 `detail.school`、`detail.hospital`、`detail.commercial` 等设施邻近度字段：

- 全局 Moran's I：判断整体空间自相关。
- Getis-Ord Gi*：判断每个网格是否属于热点或冷点。

如果 PySAL 相关依赖不可用，会使用 NumPy 实现的 fallback 版本，因此缺少 `esda` 不会阻止项目启动。

#### 服务区/等时圈：`service_area.py`

服务区分析同样基于路网图：

1. 将设施中心吸附到最近路网节点。
2. 按给定时间阈值，例如 5、10、15 分钟，计算可达节点。
3. 将可达节点转换成等时圈 Polygon。
4. 返回多个按 `minutes` 标注的 Feature。

#### 洪水模拟：`flood.py`

洪水模拟基于水体、研究区网格和高程数据：

1. 加载水系数据，构造起淹水体。
2. 优先读取 `dem.npy`，没有真实 DEM 时生成程序化地形。
3. 按水位筛选低于水位且连通的区域。
4. 返回淹没水面 `surface`、危险区 `hazard` 和统计信息 `meta`。

洪水动画接口会多次调用单帧模拟，返回从低水位到目标水位的多帧结果。

#### AI 助手：`ai_agent.py`

AI 助手把分析能力注册为 function calling 工具，例如：

- `assess_value`
- `site_selection`
- `route`
- `evacuate`
- `hotspot`
- `isochrone`
- `flood`

执行流程：

1. 前端发送自然语言、当前城市、地图点击上下文点。
2. 后端将用户问题和工具列表发送给 DeepSeek。
3. DeepSeek 选择要调用的工具和参数。
4. 后端通过 `_dispatch()` 调用真实分析函数。
5. 返回自然语言回复、工具名、参数、分析图层和执行步骤。

如果未配置 DeepSeek Key，`_mock_plan()` 会用关键词规则选择分析工具，保证 AI 面板仍可演示。

## 3. 前端实现

前端入口是 `frontend/src/main.js`，主组件是 `frontend/src/App.vue`。

前端职责包括：

1. 初始化 Cesium 三维地图。
2. 管理当前研究区、底图、地形、光照、点选状态。
3. 调用后端 API 获取数据和分析结果。
4. 将后端返回的 GeoJSON 渲染成三维图层。
5. 管理控制面板、结果面板和 AI 对话面板。

### 3.1 Vite 和 Cesium 配置

`frontend/vite.config.js` 使用三个关键配置：

```js
plugins: [vue(), cesium()],
server: {
  port: 5173,
  proxy: {
    '/api': 'http://localhost:8000'
  }
}
```

含义：

- `vue()`：支持 Vue 单文件组件。
- `vite-plugin-cesium`：自动处理 Cesium 的 Workers、Assets、Widgets 等静态资源。
- `/api` 代理：前端请求 `/api/cities` 时，开发服务器会转发到 `http://localhost:8000/api/cities`。

因此开发期前端代码不用写死后端域名，也避免浏览器跨域问题。

### 3.2 API 封装

所有后端调用封装在 `frontend/src/api/index.js`：

```js
const http = axios.create({
  baseURL: import.meta.env.VITE_API_BASE || '',
  timeout: 120000
})
```

默认 `baseURL` 为空，所以请求会发给当前前端域名，例如：

```js
http.get('/api/cities')
```

在开发环境中，这个请求由 Vite 代理到后端。生产环境如果需要独立 API 域名，可以设置 `VITE_API_BASE`。

API 封装对组件暴露清晰的方法：

```js
api.cities()
api.pois(city)
api.value(city, weights, resolution)
api.site(city, min_score, top_k, weights, resolution)
api.hotspot(city, weights, resolution, k, attr)
api.route(city, start, end, optimize, hazard, mode, vias)
api.flood(city, water_level, resolution)
api.ai(prompt, city, context_point, session_id)
```

组件不直接写 axios 请求，只调用这些方法。

### 3.3 App.vue 主流程

`App.vue` 是前端的状态和流程中心。

页面挂载时执行：

```js
const meta = await api.cities()
this.cities = meta.cities
this.poiCn = meta.poi_cn
this.currentCity = meta.default

this.viewer = await createViewer(this.$refs.cesiumEl)
this.lm = new LayerManager(this.viewer)
this.tileset = await loadOsmBuildings(this.viewer)
flyToCity(this.viewer, this.cities[this.currentCity])
```

也就是说，前端启动后先从后端拿研究区元数据，再初始化 Cesium Viewer、图层管理器和 OSM Buildings。

用户点击地图时，`onMapClick()` 将经纬度传回 `App.vue`。根据当前 `pendingPick` 状态，点击点可能被保存为：

- 路线起点
- 路线终点
- 撤离起点
- 服务区中心
- 三维视域观察点
- 途径点

### 3.4 控制面板到后端的调用链

左侧 `ControlPanel` 不直接调用后端。它通过事件通知 `App.vue`：

```vue
<ControlPanel @run="onRun" ... />
```

`App.vue` 的 `onRun()` 根据动作名调用对应方法：

```js
async onRun(action) {
  this.busy = true
  try { await this[action.fn](action.payload || {}) }
  finally { this.busy = false }
}
```

例如用户点击“价值评估”：

```text
ControlPanel
  -> emit('run', { fn: 'runValue', payload })
  -> App.vue onRun()
  -> App.vue runValue()
  -> api.value()
  -> POST /api/analysis/value
  -> backend value.assess_value()
  -> 返回 FeatureCollection
  -> LayerManager.renderValueGrid()
```

价值评估、选址分析和热点分析都允许前端传入网格分辨率。分辨率越高，专题面越细，但后端计算量也越大。热点分析还可以传入 `attr`：

- `score`：综合价值热点。
- `school`、`hospital`、`commercial`、`transit`、`road`、`scenic`：对应设施邻近度的分项热点。

也就是说，前端既可以演示“哪里综合价值聚集”，也可以演示“哪里学区优势聚集”“哪里医院可达性更强”等分项空间统计。

路径规划也是类似：

```text
地图点选起点/终点
  -> App.vue 保存 picks.routeStart / picks.routeEnd
  -> 用户运行路径规划
  -> api.route(city, start, end, optimize, hazard, mode, vias)
  -> POST /api/route
  -> backend routing.route()
  -> 返回 LineString FeatureCollection
  -> LayerManager.renderRoute()
```

洪水模拟和撤离路径可以联动：

```text
runFlood()
  -> POST /api/flood
  -> 返回 surface + hazard
  -> 前端渲染淹没面，并保存 this.floodHazard
  -> 如果用户选择 reroute 且已有撤离起点
  -> POST /api/evacuate，传入 hazard
  -> 后端避开危险区重新计算撤离路线
```

### 3.5 Cesium 图层渲染

Cesium 相关逻辑在 `frontend/src/cesium/`：

- `viewer.js`：创建 Viewer、切换底图、加载 OSM Buildings、相机飞行、地图点击、光照和阴影。
- `layers.js`：把后端 GeoJSON 结果渲染到 Cesium。
- `analysis3d.js`：前端本地三维视域分析。

`LayerManager` 是主要图层管理类。它负责：

- `renderValueGrid()`：渲染价值网格 Polygon。
- `renderHotspot()`：渲染热点/冷点网格。
- `renderPois()`：渲染 POI 点。
- `renderRoute()`：渲染路线 LineString。
- `renderIsochrone()`：渲染等时圈 Polygon。
- `renderFlood()`：渲染洪水淹没面。
- `marker()`：渲染地图点选标记。
- `clear()` / `clearAll()`：清理图层。

后端返回 GeoJSON，前端不需要重新计算空间分析结果，只负责将不同类型的 Feature 转成 Cesium entity 或 data source。

价值评估、选址分析和热点分析都属于网格面图层。为了避免结果面悬浮在城市建筑上方，前端不再把价值网格做 `extrudedHeight` 挤出，而是使用 Cesium polygon 的 `classificationType: Cesium.ClassificationType.TERRAIN` 贴合地球/地形表面渲染：

```js
polygon: {
  hierarchy: Cesium.Cartesian3.fromDegreesArray(ring),
  material: heatColor(score / 100, 0.38),
  classificationType: Cesium.ClassificationType.TERRAIN
}
```

热点的热点/冷点面也使用 terrain classification，并降低不透明度，避免大面积专题面遮挡影像、道路和建筑底图。这样价值评估、选址和热点结果都作为地表专题图层叠加在底图上，而不是漂浮在 OSM Buildings 白模之上；路线、POI、洪水和等时圈也分别使用 `clampToGround` 或 terrain classification 保持与球体表面对齐。

### 3.6 AI 面板交互

AI 面板是 `frontend/src/components/AIChat.vue`。

用户发送问题后：

```js
const r = await api.ai(q, this.city, this.contextPoint, this.session)
this.$emit('result', r)
```

返回结果会交给 `App.vue` 的 `onAIResult()`：

```js
onAIResult(r) {
  if (k === 'value' || k === 'site') this.lm.renderValueGrid(r.layer)
  else if (k === 'hotspot') this.lm.renderHotspot(r.layer)
  else if (k === 'route') this.lm.renderRoute(r.layer)
  else if (k === 'flood') this.lm.renderFlood(r.layer)
}
```

也就是说，AI 返回的数据结构和普通按钮触发的分析结果是同一种图层结构，前端可以复用同一套 Cesium 渲染逻辑。

## 4. 前后端交互总结

项目的交互模式可以概括为：

```text
用户操作 Vue 组件
  -> App.vue 汇总当前状态和参数
  -> frontend/src/api/index.js 发起 Axios 请求
  -> Vite dev server 将 /api 代理到 FastAPI
  -> backend/app/routers/api.py 校验请求并分发
  -> backend/app/engine/* 执行空间分析
  -> 后端返回 JSON/GeoJSON
  -> App.vue 接收结果
  -> LayerManager 渲染到 Cesium 三维地图
  -> ResultPanel 展示统计摘要
```

关键设计点：

- 前端不直接处理复杂空间分析，主要负责交互状态和三维可视化。
- 后端不关心 Cesium 渲染细节，只返回标准 JSON/GeoJSON。
- API 层通过 Pydantic 明确请求结构，便于自动校验和生成文档。
- Vite 代理让开发期请求保持相对路径 `/api`，减少跨域和环境配置问题。
- AI 助手最终仍调用同一套后端分析引擎，因此自然语言分析和按钮分析的结果格式一致。
