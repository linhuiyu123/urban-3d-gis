# 价值评估、选址分析与热点分析模块说明

本文说明本项目中的三个空间分析功能：价值评估、选址分析和热点分析。三者不是孤立功能，而是一套连续的分析流程：

```text
POI / 道路网等基础数据
  -> 价值评估：生成每个网格的综合价值分数
  -> 选址分析：从价值网格中筛选高价值地块
  -> 热点分析：判断价值或单项因子是否形成空间聚集
```

这三个功能的后端核心文件如下：

| 功能 | 接口 | 后端文件 |
|---|---|---|
| 价值评估 | `POST /api/analysis/value` | `backend/app/engine/value.py` |
| 选址分析 | `POST /api/analysis/site` | `backend/app/engine/value.py` |
| 热点分析 | `POST /api/stats/hotspot` | `backend/app/engine/stats.py` |

前端通过 `frontend/src/api/index.js` 调用接口，由 `App.vue` 接收结果，再交给 `frontend/src/cesium/layers.js` 渲染到 Cesium 地图上。

## 1. 数据基础

本模块主要使用以下空间数据：

| 数据 | 文件 | 用途 |
|---|---|---|
| POI | `data/<city>/pois.geojson` | 计算景点、商业、学校、医院、公交等设施邻近度 |
| 道路 | `data/<city>/roads.geojson` | 计算道路邻近度，也服务于路径类分析 |
| 城市配置 | `backend/app/config.py` | 提供研究区 bbox、中心点、投影坐标系 |

如果真实数据不存在，`backend/app/data_loader.py` 会自动回退到样例数据。因此项目没有真实数据时也可以运行，但真实数据会让价值评估和热点分析更有解释意义。

当前杭州主城区已经抓取了真实数据：

```text
pois: 699
roads: 72969
water: 2856
```

其中道路数据是单独的 `roads.geojson`，不是 POI 中的某一类点。价值评估里的 `road` 因子会优先使用真实道路折点计算邻近度。

## 2. 价值评估

### 2.1 功能目标

价值评估用于回答：

```text
研究区内哪些位置的综合区位价值更高？
```

后端把研究区划分为规则网格，对每个网格中心计算综合价值分数 `score`。分数范围为 `0-100`，分数越高，表示在当前权重设置下该位置越接近高价值设施或道路资源。

前端展示时会把网格渲染为贴合地表的半透明热力面：

- 暖色表示价值较高；
- 冷色表示价值较低；
- 图层贴合 Cesium 球体/地形表面，不再悬浮在城市上空；
- 透明度较低，避免遮挡底图、道路和建筑信息。

### 2.2 输入参数

接口：

```text
POST /api/analysis/value
```

请求示例：

```json
{
  "city": "hangzhou_core",
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48
}
```

参数说明：

| 参数 | 说明 |
|---|---|
| `city` | 研究区 ID，例如 `hangzhou_core` |
| `weights` | 各类因子的权重 |
| `resolution` | 网格分辨率，越大越精细，计算越慢 |

默认权重如下：

```text
scenic: 0.15
commercial: 0.25
school: 0.20
hospital: 0.15
transit: 0.20
road: 0.05
```

这些权重可以在前端调整。比如提高 `school` 权重，会让靠近学校的区域在综合价值中更占优势；提高 `commercial` 权重，则更偏向商业选址。

### 2.3 核心算法

价值评估采用“距离衰减 + 加权叠加”的方法。

第一步，根据城市 bbox 生成规则网格。每个网格中心作为一个待评价位置。

第二步，分别计算网格中心到以下评价对象的最近距离：

```text
scenic      景点/旅游资源
commercial  商业设施
school      学校
hospital    医院
transit     公共交通站点
road        道路网
```

第三步，将距离转换为邻近度：

```text
s = exp(-d / decay)
```

其中：

- `d` 是到最近对象的距离；
- `decay` 是距离衰减尺度；
- `s` 是 `0-1` 的邻近度，越近越接近 1。

第四步，对各类邻近度按权重加权：

```text
综合价值 = Σ(设施邻近度 × 权重)
```

最后将结果归一化到 `0-100`，作为每个网格的 `score`。

为了适应不同大小的研究区，后端会根据 bbox 自动调整衰减尺度。这样主城区和都市区不会因为范围不同而出现分数整体过低或过高的问题。

### 2.4 输出结果

返回结果是 GeoJSON `FeatureCollection`。每个 Feature 是一个网格 Polygon：

```json
{
  "type": "Feature",
  "geometry": {
    "type": "Polygon",
    "coordinates": [[[120.1, 30.2], [120.11, 30.2], [120.11, 30.21], [120.1, 30.21], [120.1, 30.2]]]
  },
  "properties": {
    "score": 86.4,
    "lon": 120.105,
    "lat": 30.205,
    "detail": {
      "scenic": 0.421,
      "commercial": 0.883,
      "school": 0.692,
      "hospital": 0.514,
      "transit": 0.775,
      "road": 0.934
    }
  }
}
```

`detail` 字段保留每个单项因子的邻近度。它的意义很重要：热点分析可以不只分析综合 `score`，还可以单独分析学校、医院、商业、道路等因子的空间聚集。

`meta` 中包含：

```text
city
resolution
weights
score_mean
score_max
score_p90
decay_scale
```

这些字段用于结果面板展示和报告解释。

## 3. 选址分析

### 3.1 功能目标

选址分析用于回答：

```text
在当前价值模型下，哪些地块适合作为候选选址？
```

它不是另起一套算法，而是在价值评估结果上做筛选。后端先生成价值网格，再按阈值和 Top-K 条件筛选高价值地块。

适用场景包括：

- 商业门店选址；
- 住宅区位筛选；
- 学校、医院、交通等偏好型选址；
- 从全研究区中快速找出高价值候选地块。

### 3.2 输入参数

接口：

```text
POST /api/analysis/site
```

请求示例：

```json
{
  "city": "hangzhou_core",
  "min_score": 70,
  "top_k": 10,
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48
}
```

参数说明：

| 参数 | 说明 |
|---|---|
| `min_score` | 最低价值阈值，只保留 `score >= min_score` 的网格 |
| `top_k` | 只返回分数最高的 K 个地块；为空时返回全部达标地块 |
| `weights` | 与价值评估相同的权重 |
| `resolution` | 与价值评估相同的网格分辨率 |

前端还提供“重点设施”选项。本质上是调整权重，让选址结果偏向某一类设施。例如：

```text
重点设施 = 学校    -> 更偏向学区优势地块
重点设施 = 商业区  -> 更偏向商业可达性高的地块
重点设施 = 医院    -> 更偏向医疗可达性高的地块
```

### 3.3 实现流程

选址分析流程如下：

```text
调用 assess_value()
  -> 得到完整价值网格
  -> 筛选 score >= min_score 的网格
  -> 按 score 从高到低排序
  -> 如果设置 top_k，保留前 K 个
  -> 返回筛选后的 FeatureCollection
```

因为选址复用价值评估，所以它天然继承了价值评估的权重、道路邻近度、POI 邻近度和分辨率设置。

### 3.4 输出结果

输出仍然是 GeoJSON `FeatureCollection`，但只包含符合条件的候选地块。

`meta` 中包含：

```text
city
min_score
count
qualified
top_score
resolution
weights
```

字段说明：

| 字段 | 说明 |
|---|---|
| `qualified` | 满足阈值的地块总数 |
| `count` | 当前返回给前端显示的地块数量 |
| `top_score` | 候选地块中的最高分 |

如果设置了 `top_k`，`count` 可能小于 `qualified`。

## 4. 热点分析

### 4.1 功能目标

热点分析用于回答：

```text
高价值区域是否集中成片？
低价值区域是否集中成片？
某一类设施优势是否存在明显空间聚集？
```

它不是简单找最高分地块，而是做空间统计。价值评估告诉我们“哪里高”，热点分析进一步判断“高值是不是形成空间集聚”。

热点分析支持两类对象：

```text
score              综合价值热点
detail.<category>  分项因子热点
```

分项因子包括：

```text
detail.scenic
detail.commercial
detail.school
detail.hospital
detail.transit
detail.road
```

这使得热点分析可以解释更具体的问题，例如：

```text
学校邻近度是否在某些区域集中？
医院可达性是否形成明显高值区？
商业设施优势是不是只集中在少数片区？
```

### 4.2 输入参数

接口：

```text
POST /api/stats/hotspot
```

请求示例：

```json
{
  "city": "hangzhou_core",
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48,
  "k": 8,
  "attr": "score",
  "z_threshold": 1.65
}
```

参数说明：

| 参数 | 说明 |
|---|---|
| `weights` | 生成价值网格时使用的权重 |
| `resolution` | 生成价值网格时使用的分辨率 |
| `k` | K 近邻数量，默认 8 |
| `attr` | 统计字段，默认 `score`，也可传 `school`、`hospital`、`commercial` 等 |
| `z_threshold` | 热点/冷点分类阈值，默认 1.65 |

`k` 控制局部邻域大小：

```text
k 较小  -> 更强调局部差异
k 较大  -> 更强调区域趋势
```

`z_threshold` 控制显著性：

```text
1.65  -> 约 90% 置信水平，演示更清晰
1.96  -> 约 95% 置信水平，更严格
```

杭州主城区真实数据下，综合价值面比较连续。如果固定使用 95% 阈值，综合热点可能不明显。前端默认使用 90% 阈值，是为了让课程演示中能更清楚看到主要空间聚集区，同时仍保留 95% 严格选项。

### 4.3 核心算法

热点分析分三步。

第一步，生成价值网格：

```text
value.assess_value(city, weights, resolution)
```

第二步，构建 K 近邻空间权重。每个网格用中心点代表位置，并选取最近的 K 个网格作为邻居。

第三步，计算空间统计指标：

```text
全局 Moran's I      判断整体空间自相关
局部 Getis-Ord Gi*  判断每个网格是否为热点/冷点
```

分类规则：

```text
gi_z >= z_threshold   -> 热点 hot
gi_z <= -z_threshold  -> 冷点 cold
其他                  -> 不显著 none
```

### 4.4 PySAL 与降级实现

代码优先使用 PySAL 生态：

```text
Moran's I: esda.moran.Moran
Getis-Ord Gi*: esda.getisord.G_Local
空间权重: libpysal.weights.KNN
```

如果 PySAL 相关依赖不可用，或运行时出现 NaN/inf，后端会自动降级为 NumPy 实现，保证接口仍能返回结果。这对课堂演示比较重要，因为不同电脑上的 Python 环境可能不完全一致。

### 4.5 输出结果

热点分析返回 GeoJSON `FeatureCollection`。每个网格保留原有 `score` 和 `detail`，并追加：

```text
gi_z
hot_class
```

其中：

| 字段 | 说明 |
|---|---|
| `gi_z` | 局部 Gi* 的 z 分数 |
| `hot_class` | `hot`、`cold` 或 `none` |

`meta` 中包含：

```text
city
resolution
weights
attr
k
z_threshold
confidence
moran_I
moran_p
interpretation
engine
n_hot
n_cold
```

前端会用：

- 红色表示热点；
- 蓝色表示冷点；
- 灰色表示不显著；
- 结果面板展示 Moran's I、p 值、热点数量、冷点数量、统计对象和阈值。

## 5. 前端交互与展示

左侧控制面板中，三个功能的主要交互如下：

| 功能 | 可调参数 |
|---|---|
| 价值评估 | 权重、网格分辨率 |
| 选址分析 | 重点设施、价值阈值、Top-K、网格分辨率 |
| 热点分析 | 统计对象、网格分辨率、K 近邻、显著性阈值 |

前后端调用链路：

```text
ControlPanel.vue
  -> App.vue runValue / runSite / runHotspot
  -> frontend/src/api/index.js
  -> FastAPI API
  -> backend/app/engine/value.py 或 stats.py
  -> 返回 GeoJSON
  -> LayerManager 渲染到 Cesium
  -> ResultPanel.vue 展示摘要
```

图层渲染上，价值评估、选址分析和热点分析都作为贴地专题面叠加在 Cesium 地球表面：

```js
classificationType: Cesium.ClassificationType.TERRAIN
```

这样可以避免结果图层悬浮在城市上空，也不会把建筑和底图完全遮住。

## 6. 演示建议

推荐演示顺序：

1. 先运行价值评估，展示综合价值热力面。
2. 调整权重，例如提高学校或商业权重，说明模型可解释、可配置。
3. 进入选址分析，设置阈值和 Top-K，展示候选地块筛选。
4. 进入热点分析，先看综合价值热点，再切换到学校或医院分项热点。
5. 对比 90% 和 95% 显著性阈值，说明阈值越严格，热点越少。

杭州主城区真实数据下，比较适合展示：

```text
综合价值 score      适合看整体价值格局
school             适合看教育资源聚集
hospital           适合看医疗资源聚集
commercial         适合看商业设施聚集
```

`road` 和 `transit` 在主城区中分布较密，可能更容易表现为低值边缘区或冷点，不一定会出现非常醒目的热点。这属于数据分布特征，不是接口错误。

## 7. 总结

本模块以 POI 和道路网为基础，构建了一个可解释的城市区位价值分析流程。价值评估通过距离衰减和加权叠加生成 `0-100` 的综合价值网格；选址分析在价值网格上按阈值和 Top-K 筛选候选地块；热点分析进一步使用 Moran's I 和 Getis-Ord Gi* 判断综合价值或单项因子的空间聚集。三个功能共用同一套数据、权重和网格体系，既能支持交互式演示，也能在课程报告中形成完整的方法链条。
