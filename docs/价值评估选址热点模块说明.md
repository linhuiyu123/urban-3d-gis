# 价值评估、选址分析与热点分析模块说明

本文用于说明本人负责的后端模块：价值评估、选址分析与热点/空间统计分析。该模块的核心目标是基于城市 POI 与道路网等空间数据生成地段价值网格，并在此基础上完成高价值地块筛选和价值空间格局识别。

## 1. 模块定位

本模块承担项目中的“城市空间价值建模与统计分析”任务，主要包括三个接口：

| 功能 | 接口 | 主要后端文件 |
|---|---|---|
| 地段价值评估 | `POST /api/analysis/value` | `backend/app/engine/value.py` |
| 选址分析 | `POST /api/analysis/site` | `backend/app/engine/value.py` |
| 热点/空间统计分析 | `POST /api/stats/hotspot` | `backend/app/engine/stats.py` |

三个功能不是彼此孤立的。它们之间的关系是：

```text
POI / 道路网等基础空间数据
  -> 地段价值评估，生成带 score 的价值网格
  -> 选址分析，在价值网格上筛选高分地块
  -> 热点分析，对价值网格 score 字段做空间统计
```

因此，本模块的核心成果是一个可解释的价值网格。选址分析和热点分析都建立在这个价值网格之上。

## 2. 地段价值评估

### 2.1 功能目标

地段价值评估用于衡量研究区内不同位置的综合区位价值。后端将研究区划分为规则网格，并为每个网格计算一个 `0-100` 的综合分数 `score`。分数越高，表示该地块在当前权重设置下的综合价值越高。

前端会将该结果渲染成三维热力网格：

- 颜色越暖，价值越高；
- 柱体越高，价值越高；
- 每个网格保留各类设施的分项邻近度，便于解释结果来源。

### 2.2 输入参数

接口：

```text
POST /api/analysis/value
```

请求参数：

```json
{
  "city": "hangzhou_core",
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48
}
```

参数含义：

| 参数 | 含义 |
|---|---|
| `city` | 研究区 ID，例如 `hangzhou_core` |
| `weights` | 各类设施权重，用于控制价值评估偏好 |
| `resolution` | 网格分辨率，数值越大网格越细，计算越慢 |

### 2.3 核心算法

价值评估采用“距离衰减 + 加权叠加”的思路。

首先，后端在研究区 bbox 内生成规则网格。每个网格中心作为一个待评价位置。

然后，对每个网格中心分别计算它到六类评价因子的最近距离：

```text
旅游景点 scenic
商业区 commercial
学校 school
医院 hospital
公共交通 transit
道路网 road
```

其中 `scenic`、`commercial`、`school`、`hospital`、`transit` 来自 POI 数据，`road` 来自道路网 `roads.geojson`。距离计算在 UTM 投影坐标系下完成，避免直接使用经纬度距离带来的误差。

接着，将距离转换为邻近度得分。使用的衰减函数为：

```text
s = exp(-d / decay)
```

其中：

- `d` 是网格中心到最近评价对象的距离；
- `decay` 是该类设施的影响尺度；
- `s` 是 `0-1` 之间的邻近度，距离越近值越大。

最后，对各类邻近度按权重加权求和，并归一化到 `0-100`：

```text
综合价值 = Σ(设施邻近度 × 权重)
```

为了适配不同尺度的研究区，代码中还加入了 `decay_scale`。当研究区范围较大时，衰减尺度会自动放大，避免所有网格都因为距离设施太远而接近 0。

### 2.4 输出结果

输出为 GeoJSON `FeatureCollection`。每个 Feature 是一个网格 Polygon：

```json
{
  "type": "Feature",
  "geometry": {
    "type": "Polygon",
    "coordinates": [[[...]]]
  },
  "properties": {
    "score": 86.4,
    "lon": 120.16,
    "lat": 30.27,
    "detail": {
      "scenic": 0.421,
      "commercial": 0.883,
      "school": 0.692,
      "hospital": 0.514,
      "transit": 0.775,
      "road": 0.334
    }
  }
}
```

`meta` 中包含：

```text
city
resolution
weights
score_mean
score_max
score_p90
decay_scale
```

这些字段可以用于结果面板展示，也可以用于报告中解释模型参数和统计结果。

## 3. 选址分析

### 3.1 功能目标

选址分析是在价值评估结果上的进一步筛选。它先调用同一套价值评估模型生成网格，然后根据用户设置的阈值筛选高价值地块。

该功能适用于：

- 商业选址；
- 住宅地段筛选；
- 学区/医疗/交通等偏好型选址；
- 综合区位优势地块识别。

### 3.2 输入参数

接口：

```text
POST /api/analysis/site
```

请求参数：

```json
{
  "city": "hangzhou_core",
  "min_score": 70,
  "top_k": 10,
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48
}
```

参数含义：

| 参数 | 含义 |
|---|---|
| `min_score` | 最低价值阈值，只保留高于该分数的地块 |
| `top_k` | 只返回分数最高的 K 个地块；为空时返回全部达标地块 |
| `weights` | 价值评估权重 |
| `resolution` | 价值网格分辨率 |

### 3.3 实现逻辑

选址分析的实现流程为：

```text
调用 assess_value()
  -> 得到完整价值网格
  -> 筛选 score >= min_score 的网格
  -> 按 score 从高到低排序
  -> 如果设置 top_k，则截取前 K 个
  -> 返回筛选后的 FeatureCollection
```

选址并不是另起一套模型，而是价值评估的结果查询。因此它与价值评估共享权重、分辨率和评分逻辑。

### 3.4 输出结果

输出仍然是 GeoJSON `FeatureCollection`，但只包含达标地块。

`meta` 中包含：

```text
city
min_score
count
qualified
top_score
resolution
weights
```

其中：

- `qualified` 表示满足阈值的地块总数；
- `count` 表示当前返回给前端显示的地块数量；
- 如果设置了 `top_k`，则 `count` 可能小于 `qualified`。

## 4. 热点/空间统计分析

### 4.1 功能目标

热点分析用于判断高价值地块和低价值地块在空间上是否呈现聚集分布。

它解决的问题不是“哪里分数最高”，而是：

```text
高分区域是否集中成片？
低分区域是否集中成片？
价值格局是随机分布，还是存在明显空间自相关？
```

因此，热点分析是对价值评估结果的空间统计解释。

### 4.2 输入参数

接口：

```text
POST /api/stats/hotspot
```

请求参数：

```json
{
  "city": "hangzhou_core",
  "weights": {
    "scenic": 0.15,
    "commercial": 0.25,
    "school": 0.2,
    "hospital": 0.15,
    "transit": 0.2,
    "road": 0.05
  },
  "resolution": 48,
  "k": 8
}
```

参数含义：

| 参数 | 含义 |
|---|---|
| `weights` | 生成价值网格时使用的设施权重 |
| `resolution` | 生成价值网格时使用的分辨率 |
| `k` | K 近邻空间权重中的邻居数量 |

`k` 的默认值为 `8`。数值越大，热点判断越偏向区域整体趋势；数值越小，越强调局部邻近关系。

### 4.3 核心算法

热点分析分两步：

第一步，先调用价值评估模型生成带 `score` 字段的网格。

第二步，对 `score` 做空间统计分析：

1. 构建 K 近邻空间权重；
2. 计算全局 Moran's I；
3. 计算局部 Getis-Ord Gi*；
4. 根据 Gi* 的 z 分数将每个网格分类为热点、冷点或不显著区域。

分类规则：

```text
gi_z > 1.96   -> 热点 hot
gi_z < -1.96  -> 冷点 cold
其他          -> 不显著 none
```

这里的 `1.96` 对应约 95% 置信水平。

### 4.4 PySAL 与降级实现

代码优先使用 PySAL 生态中的 `esda` 和 `libpysal`：

```text
Moran's I: esda.moran.Moran
Getis-Ord Gi*: esda.getisord.G_Local
空间权重: libpysal.weights.KNN
```

如果运行环境中没有这些库，或 PySAL 在数值退化情况下返回 NaN/inf，后端会自动降级到 NumPy 实现，保证接口仍然可用。

这点对项目演示很重要，因为它降低了环境依赖风险。

### 4.5 输出结果

热点分析输出为 GeoJSON `FeatureCollection`。每个网格会在原有属性基础上追加：

```text
gi_z
hot_class
```

其中：

- `gi_z` 是局部 Gi* 的 z 分数；
- `hot_class` 取值为 `hot`、`cold` 或 `none`。

`meta` 中包含：

```text
city
resolution
weights
attr
k
moran_I
moran_p
interpretation
engine
n_hot
n_cold
```

这些字段既可以用于前端结果面板，也可以直接用于报告中的统计结果说明。

## 5. 与前端的交互方式

前端通过 `frontend/src/api/index.js` 调用后端接口。开发环境下，前端请求 `/api/...`，由 Vite 代理到后端 `http://localhost:8000`。

当前交互链路为：

```text
ControlPanel.vue
  -> App.vue 的 runValue / runSite / runHotspot
  -> frontend/src/api/index.js
  -> FastAPI 接口
  -> 后端分析引擎
  -> 返回 GeoJSON
  -> cesium/layers.js 渲染图层
  -> ResultPanel.vue 展示统计摘要
```

前端主要依赖以下字段：

```text
score
detail
gi_z
hot_class
meta.score_mean
meta.score_max
meta.score_p90
meta.min_score
meta.qualified
meta.count
meta.top_score
meta.moran_I
meta.moran_p
meta.n_hot
meta.n_cold
meta.engine
```

因此，后端在迭代时需要保持这些字段稳定，避免破坏前端渲染。

## 6. 本次改进

本次主要改进是让接口参数更加一致，并增强结果可解释性。

### 6.1 选址接口补充分辨率参数

之前选址分析虽然复用了价值评估逻辑，但接口没有显式接收 `resolution`。这会导致前端如果调整价值评估分辨率，选址分析仍然可能按默认分辨率计算。

现在 `POST /api/analysis/site` 支持：

```text
resolution
```

并在结果 `meta` 中返回实际使用的 `weights` 和 `resolution`。

### 6.2 热点接口补充权重、分辨率和 K 近邻参数

之前热点分析只接收 `city`，会用默认权重和默认分辨率重新生成价值网格。

现在 `POST /api/stats/hotspot` 支持：

```text
weights
resolution
k
```

这样热点分析可以与用户当前的价值评估参数保持一致。

### 6.3 热点结果保留价值网格元数据

热点分析结果的 `meta` 现在会保留价值评估网格的元数据，包括：

```text
city
resolution
weights
score_mean
score_max
score_p90
decay_scale
```

这有助于解释热点结果是基于哪一套价值模型计算出来的。

## 7. 给前端同学的接入建议

目前后端改动是向后兼容的。前端不改也能继续运行，接口会使用默认参数。

但为了让展示逻辑更加一致，建议前端后续做以下小改动：

1. 选址分析调用 `/api/analysis/site` 时，把当前价值评估面板的 `valueRes` 一起传给后端。
2. 热点分析调用 `/api/stats/hotspot` 时，把当前 `weights`、`valueRes` 一起传给后端。
3. 如果不想新增界面控件，热点分析的 `k` 可以先固定为 `8`。

建议的前端请求封装形式：

```js
site: (city, min_score, top_k, weights, resolution) =>
  http.post('/api/analysis/site', { city, min_score, top_k, weights, resolution }).then(r => r.data),

hotspot: (city, weights, resolution, k = 8) =>
  http.post('/api/stats/hotspot', { city, weights, resolution, k }).then(r => r.data),
```

这样可以保证：

```text
价值评估、选址分析、热点分析使用同一套权重和网格分辨率。
```

## 8. 可放入报告的总结表述

本模块实现了基于 POI 邻近度的城市地段价值建模，并在此基础上完成选址筛选和空间统计分析。价值评估模块将研究区划分为规则网格，计算每个网格到景点、商业区、学校、医院、公共交通和道路等设施的距离，通过指数距离衰减函数转化为邻近度得分，再按权重加权叠加得到 `0-100` 的综合价值分数。选址分析复用该价值网格，根据阈值和 Top-K 条件筛选高价值地块。热点分析进一步对价值网格的 `score` 字段进行全局 Moran's I 和局部 Getis-Ord Gi* 统计，识别高值聚集区、低值聚集区和不显著区域。该模块通过 GeoJSON 向前端输出标准化空间结果，便于 Cesium 三维地图渲染和结果面板展示。
